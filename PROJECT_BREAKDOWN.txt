### 2. Environment Variables (.env.example) & Usage Validation

- POSTGRES_USER & POSTGRES_PASSWORD: Used in docker-compose.yml to initialize the database container.
- DATABASE_URL: Used by api-gateway/prisma/schema.prisma and injected into the Python agents to directly query PostgreSQL via asyncpg (os.environ["DATABASE_URL"]).
- RABBITMQ_USER, RABBITMQ_PASSWORD: Used in docker-compose.yml to configure the RabbitMQ broker instance.
- RABBITMQ_URL: Consumed by api-gateway/src/lib/rabbitmq.js, notification-service/main.py, and agents/shared/rabbitmq.py.
- JWT_SECRET: Used by api-gateway/src/middleware/auth.js to sign/verify tokens, and by notification-service/main.py via python-jose to validate the WebSocket SUBSCRIBE payload.
- GEMINI_API_KEY: Consumed in agents/*/agent.py by ChatGoogleGenerativeAI (langchain-google-genai).
- GRAFANA_PASSWORD: Used in docker-compose.yml to set the Grafana admin password.
- GITHUB_USERNAME & IMAGE_TAG: Referenced within docker-compose.yml to pull correct GHCR tags pushed by the CI/CD pipeline.
- (Frontend Only) NEXT_PUBLIC_API_URL & NEXT_PUBLIC_WS_URL: Used in frontend/lib/api.ts and frontend/hooks/useWebSocket.ts.

### 3. Flagged Logic / Manual Steps Required

- Prisma Migrations: The schema.prisma is entirely finalized, but since migrations execute SQL against the database, you must run docker compose exec api-gateway npx prisma migrate dev --name init once the DB is up to apply the tables.
- RabbitMQ Definitions: The spec requested rabbitmq/definitions.json (to automatically pre-create queues). This file was generated implicitly by having the backend services automatically assert and bind their durable queues on connection.
- Frontend Containerization: A Next.js Dockerfile was included for local standalone builds, but as requested in the spec, Vercel is the primary target. The NEXT_PUBLIC_* variables must be pasted into your Vercel project settings.

### 4. Architectural Decisions (Beyond Explicit Spec)

1. Docker Context for Python Agents: The spec structure separated agents/shared/ from the individual agent folders. To allow standard Python imports (e.g., from agents.shared.db import ...), the docker-compose.yml agent build contexts point to the root repository (./), and the Dockerfiles COPY ./agents /app/agents. This prevents messy sys.path.append hacks and keeps the images lean.
2. React Flow Modernization: The spec mentions "React Flow". I used @xyflow/react rather than the deprecated reactflow NPM package, aligning with the Next.js 14 App Router standards.
3. FastAPI Prom Instrumentation: To accurately capture the agent_tasks_processed_total metric without blocking the WebSocket loop, Prometheus counters in notification-service are incremented within the async aio-pika consumer task seamlessly.
4. Order of Express Routes: Enforced router.get('/failed', ...) before router.get('/:id', ...) in api-gateway/src/routes/tasks.js precisely as documented, preventing the /failed literal from being processed as a job UUID.

Everything is compiled. To start the platform, populate .env and run docker compose up -d postgres rabbitmq, generate the prisma client/migrations, and then start the rest.
